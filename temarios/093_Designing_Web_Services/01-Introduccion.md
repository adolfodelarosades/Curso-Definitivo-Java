# Capítulo 1 Introducción

El ámbito de los Web services- servicios web (componentes de software a los que se puede acceder mediante programación a través de protocolos estándar de Internet) se está expandiendo rápidamente debido a la creciente ***necesidad de comunicación e interoperabilidad entre aplicaciones***. Los servicios web exponen una interfaz estándar que es independiente de la plataforma y la tecnología. Al cumplir con los estándares aceptados en toda la industria, ***los servicios web proporcionan un medio de comunicación entre las aplicaciones de software que se ejecutan en diferentes plataformas y están escritas en diferentes lenguajes de desarrollo de aplicaciones y que presentan al usuario información dinámica basada en el contexto***.

Desde su exitosa introducción en ***1999***, la plataforma **J2EE** se ha convertido en un estándar integrado para implementar y distribuir aplicaciones empresariales portátiles y distribuidas. Un factor importante que contribuye a este éxito es que la plataforma **J2EE** se ha diseñado a través de un proceso abierto, el **Java Community Process (JCP)**. Este proceso abierto ha involucrado a una variedad de proveedores de computación empresarial para garantizar que la plataforma cumpla con el espectro más amplio posible de requisitos de aplicaciones empresariales. Como resultado, la plataforma **J2EE** aborda los problemas centrales que impiden los esfuerzos de las organizaciones para mantener un ritmo competitivo en la economía de la información. Las organizaciones han reconocido esto y rápidamente adoptaron el estándar de la plataforma **J2EE**.

La plataforma **J2EE** ha evolucionado aún más para adaptarse a la creciente popularidad y uso de los servicios web. La plataforma ha agregado soporte para tecnologías y componentes específicos de servicios web. ***Los estándares de servicios web garantizan la interoperabilidad entre plataformas y lenguajes de programación***. También la portabilidad, la capacidad de migrar aplicaciones de una plataforma a otra, es igual de importante. Significativamente, la plataforma **J2EE** agrega portabilidad a los servicios web. Al combinar portabilidad e interoperabilidad, la plataforma **J2EE** es una plataforma convincente para desarrollar sistemas distribuidos. Estos nuevos componentes y tecnologías **J2EE** cumplen con los requisitos de la mayoría de las empresas: estas tecnologías permiten a las empresas exponer sus aplicaciones existentes como servicios web y desarrollar e implementar nuevos servicios web. Esta evolución de servicios web de la plataforma **J2EE** se ha logrado con el mismo proceso abierto que antes, involucrando a los principales proveedores de computación empresarial. Ha asegurado que la plataforma **J2EE** sea la plataforma elegida para desarrollar y desplegar aplicaciones empresariales y servicios Web portátiles e interoperables.

Veamos algunas de las razones por las que esto es así. La plataforma **J2EE** ha agregado sus funciones de servicios web a una plataforma que ya ha estandarizado el desarrollo y la implementación de aplicaciones empresariales portátiles. Con estas nuevas funciones, la plataforma **J2EE** ofrece a las empresas los siguientes beneficios:

* Permite que las aplicaciones empresariales existentes se amplíen como servicios web de manera estándar, fácil, portátil e interoperable

* Ayuda a extender el alcance de las aplicaciones empresariales existentes y nuevas a un nuevo conjunto de clientes más allá de la ya amplia variedad de clientes admitidos por las plataformas **J2EE** anteriores.

* Permite el uso de tecnologías robustas y comercialmente probadas (como enterprise bean components, servlets, etc.) como la columna vertebral para desarrollar nuevos servicios.

* Ayuda a que los sistemas de información empresarial - Enterprise Information Systems (EIS) existentes se integren a los usuarios finales de forma portátil

Este libro trata sobre el desarrollo de servicios web sólidos y efectivos en la plataforma Java TM 2, Enterprise Edition (J2EE TM ) - Web services on the JavaTM 2 Platform, Enterprise Edition (J2EETM), versión 1.4 o posterior. El libro no solo describe las tecnologías utilizadas para los servicios web, sino que también presenta pautas para usar estas tecnologías para diseñar y desarrollar servicios web. Muchas de estas pautas se han formulado a partir de nuestra propia experiencia en el desarrollo de servicios web y, como tales, las hemos ilustrado con ejemplos reales cuando ha sido posible.

Este primer capítulo sienta las bases para el libro: le da un vistazo a algunos de los conceptos clave que subyacen a los detalles arquitectónicos de los servicios web y proporciona motivaciones para desarrollar aplicaciones basadas en servicios web. Destaca los beneficios de los servicios web y explica por qué tiene sentido que las aplicaciones distribuidas utilicen servicios web como su arquitectura subyacente.

## 1.1. ¿Qué son los servicios web?

En este punto, es posible que se pregunte qué son exactamente los Web services y, lo que es más importante, por qué su empresa debería considerar hacer que algunas de sus propias funciones estén disponibles como un Web services. Se dan numerosas definiciones para los Web services, que van desde las más técnicas hasta las más simples. Por ejemplo, la organización World Wide Web Consortium (W3C), que establece los estándares para los Web services, los define de la siguiente manera: ***“Un Web services es un sistema de software identificado por un URI cuyas interfaces y enlaces públicos se definen y describen mediante XML. Su definición puede ser descubierta por otros sistemas de software. Estos sistemas pueden luego interactuar con el Web services de la manera prescrita por su definición, utilizando mensajes basados en XML transmitidos por protocolos de Internet”***. Una definición más simple, y quizás más útil, podría ser: ***“Un Web services es una aplicación de software, accesible en la Web (o en la intranet de una empresa) a través de una URL, a la que acceden los clientes mediante protocolos basados en XML, como el Simple Object Access Protocol (SOAP), enviado a través de protocolos de Internet aceptados, como HTTP. Los clientes acceden a una aplicación de Web services a través de sus interfaces y enlaces(bindings), que se definen mediante XML artifacts - artefactos XML, como un archivo Web Services Definition Language (WSDL) - Lenguaje de Definición de Servicios Web (WSDL)”***.

Los Web services son el resultado de la evolución natural de la Web. Inicialmente, la Web constaba de sitios que eran páginas HTML simples. Posteriormente, las aplicaciones web generaron dinámicamente estas mismas páginas HTML. Por ejemplo, un sitio web de mapas inicialmente solo proporcionaba enlaces estáticos a mapas de varias ciudades y lugares. Posteriormente, este mismo sitio web de mapas se convirtió en una aplicación web de mapas que proporcionaba direcciones de conducción, mapas personalizados, etc. A pesar de sus capacidades ampliadas, las aplicaciones web todavía están limitadas a las capacidades de GUI restringidas de sus páginas HTML: una aplicación web solo se puede usar a través de la GUI limitada vinculada a las páginas HTML. ***Los Web services van más allá de esta limitación, ya que separan el sitio web o la aplicación (el servicio) de su GUI HTML. En su lugar, el servicio se representa en XML y está disponible a través de la Web como XML.***  Como resultado, el mismo sitio web de mapas puede ampliar su funcionalidad para proporcionar un Web services que otras empresas pueden utilizar para proporcionar indicaciones para llegar a sus propias oficinas, integrarse con sistemas de posicionamiento global, etc.

Los Web services, o simplemente services(servicios), se basan en los conocimientos adquiridos en entornos informáticos distribuidos más maduros (como **CORBA** y la invocación de métodos remotos de Java) ***para permitir la comunicación y la interoperabilidad entre aplicaciones***. ***Los Web services proporcionan una forma estandarizada para que las aplicaciones expongan su funcionalidad en la web o se comuniquen con otras aplicaciones en una red, independientemente de la implementación, el lenguaje de programación o la plataforma informática de la aplicación***.

***¿Por qué una empresa debería considerar la implementación de aplicaciones basadas en Web services?*** Los Web services pueden proporcionar un medio para que una empresa amplíe sus ofertas comerciales, aumente la eficiencia de su procesamiento comercial y mejore la experiencia del cliente. Al incluir con sus propios servicios las ofertas de múltiples socios, tanto la empresa como los socios amplían sus capacidades y su base de negocios. Los Web services no solo pueden ayudar a automatizar el procesamiento comercial, sino que también pueden agilizar las interacciones con servicios externos, como tarjetas de crédito y servicios de envío. ***Como resultado, a los clientes se les ofrece una experiencia enriquecida: más opciones y más elecciones, junto con más flexibilidad***.

Como cualquier aplicación, ***las aplicaciones basadas en Web services pueden realizar una variedad de funciones***. Algunos pueden manejar solo solicitudes simples de información, mientras que otros pueden implementar procesos e interacciones comerciales complejos. Mientras que las aplicaciones basadas en navegador se ocupan de la representación de datos para los usuarios finales, los  Web services permiten que los clientes utilicen la web mediante programación no solo para obtener información, sino también para acceder a estos componentes del servicio y su funcionalidad. Además, las aplicaciones pueden incorporar la funcionalidad del Web services para su propio uso.

* ***Quizás la razón más importante del aumento del uso de los Web services (la fuerza principal para su adopción generalizada) es que los Web services promueven la interoperabilidad entre diferentes plataformas, sistemas e languages***.

* ***El uso de Web services también está aumentando porque reduce los costos operativos al permitir que las organizaciones amplíen y reutilicen la funcionalidad de su sistema existente***.

***Un Web services permite una arquitectura orientada a servicios***, que es un estilo arquitectónico que ***promueve la reutilización del software mediante la creación de servicios reutilizables***. ***Las arquitecturas tradicionales orientadas a objetos promueven la reutilización mediante la reutilización de clases u objetos***. Sin embargo, los objetos suelen tener un grano demasiado fino para una reutilización eficaz. Por lo tanto, surgieron ***arquitecturas orientadas a componentes que utilizan componentes de software como entidades reutilizables***. Estos componentes consisten en un conjunto de clases relacionadas, sus recursos e información de configuración. Las arquitecturas orientadas a componentes siguen siendo una forma poderosa de diseñar sistemas de software; sin embargo, no abordan los problemas adicionales que surgen de los entornos empresariales actuales. Hoy en día, los entornos empresariales son bastante complejos debido al uso de una variedad de plataformas de software y hardware, comunicación distribuida basada en Internet, integración de aplicaciones empresariales, etc. ***Las arquitecturas orientadas a servicios abordan estos problemas utilizando un servicio como una entidad reutilizable***. ***Los servicios*** suelen ser de grano más grueso que los componentes y ***se centran en la funcionalidad proporcionada por sus interfaces***. ***Estos servicios se comunican entre sí y con los clientes de los usuarios finales a través de interfaces bien definidas y conocidas***. ***La comunicación puede variar desde un simple paso de mensajes entre los servicios hasta un escenario complejo en el que un conjunto de servicios se coordinan entre sí para lograr un objetivo común***. Estas arquitecturas permiten a los clientes, a través de la red, invocar la funcionalidad de un Web services a través de las interfaces expuestas del servicio.

En una arquitectura orientada a servicios, tiene lo siguiente:

* ***Un servicio que implementa la lógica empresarial y expone esta lógica empresarial a través de interfaces bien definidas***.

* ***Un registro donde el servicio publica sus interfaces para permitir que los clientes descubran el servicio***.

* ***Clientes (incluidos los clientes que pueden ser servicios en sí mismos) que descubren el servicio utilizando los registros y acceden al servicio directamente a través de las interfaces expuestas***.

Una ventaja importante de una arquitectura orientada a servicios es que permite el desarrollo de aplicaciones débilmente acopladas que pueden distribuirse y son accesibles a través de una red. Para habilitar esta arquitectura, necesita lo siguiente:

* Un mecanismo que permite a los clientes acceder a un servicio y registro.

* Un mecanismo para permitir que diferentes servicios registren su existencia con un registro y una forma para que los clientes consulten el registro de servicios disponibles. Los servicios web se basan en una arquitectura en la que el servicio se puede ubicar en la red y su ubicación es transparente, lo que significa que los clientes pueden descubrir dinámicamente un servicio en particular que desean utilizar.

* Un mecanismo para que los servicios expongan interfaces bien definidas y una forma para que los clientes accedan a esas interfaces.

Un punto importante a tener en cuenta es que los Web services están fuertemente respaldados por prácticamente todos los jugadores clave en el campo de la web. Este respaldo garantiza virtualmente que la tecnología de Web services está aquí para quedarse, y su aceptación será más generalizada.

A diferencia de los entornos distribuidos tradicionales, los Web services enfatizan la interoperabilidad. Los Web services son independientes de un lenguaje de programación en particular, mientras que los entornos tradicionales tienden a estar ligados a un lenguaje u otro. Del mismo modo, dado que pueden vincularse fácilmente a diferentes mecanismos de transporte, los Web services ofrecen más flexibilidad en la elección de estos mecanismos. Además, a diferencia de los entornos tradicionales, los Web services a menudo no están vinculados a frameworks de trabajo de clientes o servidores particulares. En general, los Web services se adaptan mejor a un conjunto de relaciones de granularidad gruesa y débilmente acopladas. Confiar en XML brinda a los servicios web una ventaja adicional, ya que XML hace posible el uso de documentos en entornos heterogéneos.

Los Web services, al basarse en los estándares web existentes, se pueden utilizar sin necesidad de cambios en la infraestructura web. Sin embargo, si bien pueden ser más amigables con el firewall que los entornos informáticos tradicionales, los Web services tienden a no ser tan eficientes en términos de procesamiento de espacio y tiempo.

## 1.2. Beneficios de los servicios web

Los Web services están ganando popularidad debido a los beneficios que brindan. A continuación se enumeran algunos de los beneficios clave:

Interoperabilidad en un entorno heterogéneo : probablemente el beneficio clave del modelo de servicio web es que permite que diferentes servicios distribuidos se ejecuten en una variedad de plataformas y arquitecturas de software, y permite que se escriban en diferentes lenguajes de programación. A medida que las empresas se desarrollan con el tiempo, agregan sistemas y soluciones que a menudo requieren diferentes plataformas y con frecuencia no se comunican entre sí. Posteriormente, quizás debido a una consolidación o la adición de otra aplicación, se hace necesario unir esta funcionalidad dispar. La mayor fortaleza de los servicios web es su capacidad para permitir la interoperabilidad en un entorno heterogéneo. Siempre que los diversos sistemas estén habilitados para los servicios web, pueden utilizar los servicios para interoperar fácilmente entre sí.

Servicios comerciales a través de la Web : una empresa puede utilizar los servicios Web para aprovechar las ventajas de la World Wide Web para sus operaciones. Por ejemplo, una empresa podría poner su catálogo de productos e inventario a disposición de sus proveedores a través de un servicio web para lograr una mejor gestión de la cadena de suministro.

Integración con los sistemas existentes : la mayoría de las empresas tienen una enorme cantidad de datos almacenados en los sistemas de información empresarial existentes, y el costo de reemplazar estos sistemas es tal que descartar estos sistemas heredados puede no ser una opción. Los servicios web permiten a los desarrolladores de aplicaciones empresariales reutilizar e incluso comercializar estos activos de información existentes. Los servicios web brindan a los desarrolladores formas estándar de acceder a servicios de nivel intermedio y de back-end, como sistemas de administración de bases de datos y monitores de transacciones, e integrarlos con otras aplicaciones. Además, debido a que estos servicios se brindan de manera consistente, los desarrolladores no necesitan aprender nuevos modelos o estilos de programación a medida que se expanden las necesidades de integración.

Libertad de elección : los estándares de servicios web han abierto un gran mercado para herramientas, productos y tecnologías. Esto brinda a las organizaciones una amplia variedad de opciones y pueden seleccionar las configuraciones que mejor se adapten a los requisitos de su aplicación. Los desarrolladores pueden mejorar su productividad porque, en lugar de tener que desarrollar sus propias soluciones, pueden elegir entre un mercado listo de componentes de aplicaciones listos para usar. Además, las herramientas brindan la capacidad de moverse rápida y fácilmente de una configuración a otra según sea necesario. Los servicios web también garantizan la estandarización de las herramientas, de modo que las herramientas de desarrollo puedan adoptar nuevas herramientas, ya sea de proveedores de servidores o desarrolladores de herramientas de terceros, según surjan las necesidades.

Admitir más tipos de clientes : dado que un objetivo principal de los servicios web es mejorar la interoperabilidad, exponer las aplicaciones o servicios existentes como servicios web aumenta su alcance a diferentes tipos de clientes. Esto ocurre independientemente de la plataforma en la que se base el cliente: no importa si el cliente se basa en las plataformas Java o Microsoft o incluso si se basa en una plataforma inalámbrica. En resumen, un servicio web puede ayudarlo a extender sus aplicaciones y servicios a un amplio conjunto de tipos de clientes.

Los servicios web, al crear un estándar de programación común, ayudan a mejorar la productividad de la programación. ## 1.3. Desafíos del desarrollo de servicios web
Los desarrolladores de servicios web enfrentan numerosos desafíos. Los servicios web son un nuevo paradigma y, como tales, están lejos de ser maduros. Debido a que se encuentran en sus etapas iniciales, las tecnologías de servicios web están evolucionando rápidamente y, a veces, en direcciones inesperadas. Independientemente de la plataforma que elija usar un desarrollador, existen algunos desafíos específicos para desarrollar servicios web. Por un lado, las tecnologías y los estándares están evolucionando. Como resultado, los desarrolladores pueden esperar que estas tecnologías cambien a medida que se amplíen para brindar un soporte de servicios web mejorado. Si bien estos cambios pueden causar algunas dificultades al principio, a medida que pasa el tiempo, los desarrolladores deberían descubrir que el uso de tecnologías como las que se ofrecen con la plataforma J2EE 1.4 reducirá los desafíos para el desarrollo de servicios web.

Es importante tener en cuenta estos factores al desarrollar servicios web.

1.3.1. Tecnologías y productos en evolución
Un desafío clave en la creación de soluciones basadas en servicios web hoy en día es que los servicios web todavía están en pañales. Los servicios web, debido a su énfasis en la interoperabilidad entre plataformas, sistemas operativos y lenguajes de programación, se basan en una colección de tecnologías y varios estándares y especificaciones, muchos de los cuales aún se están definiendo y perfeccionando. Actualmente se están definiendo nuevos estándares adicionales para permitir que los servicios Web alcancen todo su potencial.

Los productos y tecnologías de servicios web emergentes son muy prometedores y, a medida que se formalizan los estándares y se incorporan proveedores adicionales, su promesa se está cumpliendo. Han comenzado a aparecer más y más productos y tecnologías que implementan las especificaciones del servicio web, y están teniendo más éxito.

Las empresas suelen utilizar los servicios web como un medio para distribuir datos o información. Además, muchas empresas utilizan servicios para realizar transacciones comerciales. Dichas transacciones comerciales pueden requerir un servicio para acceder a otros servicios; en cierto sentido, para realizar una transacción global. Aunque está en proceso de definición, actualmente no existen estándares universalmente aceptados para este tipo de transacciones globales.

La interoperabilidad es un desafío continuo. Los servicios web ya han alcanzado un grado significativo de interoperabilidad, pero se necesitan más estándares para el tipo de interoperabilidad generalizada que cumple con la visión de los servicios web.

Un desafío adicional con los servicios web es la coordinación de múltiples servicios para procesar la lógica empresarial. A menudo, lo que aparece para el usuario final como un único proceso comercial, en realidad se implementa como una serie de etapas en un flujo de trabajo, y cada etapa del flujo de trabajo puede implementarse como un servicio independiente. En tales casos, todos los servicios deben coordinarse entre sí durante los distintos pasos del procesamiento de la lógica empresarial para lograr el objetivo deseado. Los estándares son necesarios para la coordinación entre los servicios. Dichos estándares están en proceso de definición, pero ninguno ha sido aún universalmente aceptado. De manera similar a los mencionados anteriormente, actualmente se están definiendo varios otros estándares para otras áreas, como la seguridad, etc.

1.3.2. Seguridad
No debería sorprender a nadie, pero la seguridad es tan importante para los servicios web como lo es para otras aplicaciones empresariales. De hecho, ahora que las aplicaciones en la Web abren los procesos y datos comerciales de una empresa a clientes distribuidos, la seguridad se convierte en un factor aún más importante.

La seguridad de los sistemas de información, si bien es de suma importancia para los administradores de TI y los arquitectos de sistemas, también debe manejarse con el equilibrio adecuado. No solo es importante proteger los activos de información para maximizar su valor, sino que se debe tener cuidado de no sobreproteger estos activos hasta el punto de poner en peligro ese mismo valor. Tradicionalmente, los departamentos de TI han mantenido un nivel de control relativamente alto, si no completo, sobre el entorno de servidores y clientes. Cuando los activos de información están expuestos en entornos menos protegidos, como a través de servicios web, se vuelve cada vez más importante mantener una seguridad estricta sobre los activos más sensibles, al tiempo que permite un acceso relativamente más fácil a otros activos.

Una de las dificultades en el manejo de sistemas dispares es proporcionar un modelo de seguridad integrado. La seguridad debe ser compatible con los mecanismos existentes. En los casos en que los clientes necesiten acceder a información segura, los mecanismos deben mantener un alto nivel de seguridad (y la confianza del usuario) sin dejar de ser tan discretos y transparentes como sea posible.

Los problemas clave de seguridad para los servicios web se relacionan con la autenticación, la autorización y la garantía de la confidencialidad. Los estándares de servicios web para la seguridad son un área de alta prioridad para la comunidad, pero, como todavía están evolucionando, siguen siendo un trabajo en progreso. El Capítulo 7 examina estos problemas y proporciona algunas soluciones provisionales.

1.3.3. Confiabilidad, disponibilidad y escalabilidad
Los servicios web suelen tratarse de aplicaciones distribuidas a gran escala. Con este tipo de aplicaciones, la confiabilidad, disponibilidad y escalabilidad de los servicios de la plataforma subyacente y sus tecnologías son consideraciones importantes.

La confiabilidad es el aspecto de un servicio web que representa qué tan bien mantiene su servicio y la calidad del servicio. A menudo, la confiabilidad se mide por la cantidad de fallas que ocurren en un período de tiempo determinado. Para los servicios web, la confiabilidad puede ser más difícil de lograr debido a la naturaleza poco confiable del transporte subyacente, HTTP; HTTP proporciona solo la entrega de mejor esfuerzo y no garantiza la entrega de paquetes o en orden. Un servicio web se considera más confiable cuanto más fácil y automáticamente pueda manejar los cambios en los patrones de uso y las configuraciones del sistema.

La disponibilidad se refiere a si el servicio web está presente o listo para su uso inmediato. ¿Un cliente tiene que esperar por el servicio web o la solicitud del cliente se maneja de inmediato? En otras palabras, la disponibilidad representa la probabilidad de que un servicio esté disponible.

Los servicios web que escalan de manera efectiva pueden manejar fácilmente una gran cantidad de interacciones con los clientes. Para dichos servicios web, la plataforma y las tecnologías deben administrar de manera eficiente los recursos y servicios del sistema (como las conexiones y transacciones de la base de datos). La plataforma también debe poder manejar el análisis XML, que se realiza para validar el contenido de un documento. El análisis de XML es un proceso intensivo en computación que afecta significativamente el rendimiento. XML por sí mismo implica un formato muy detallado que aumenta significativamente el tamaño de la carga útil en comparación con un formato binario equivalente.

Para lograr confiabilidad, disponibilidad y escalabilidad, los servicios web no solo deben ser lo suficientemente flexibles para ejecutarse en cualquier configuración de servidor adecuada para los volúmenes de clientes anticipados, sino que también deben poder cambiar configuraciones fácilmente cuando sea necesario. El soporte de una plataforma para entornos de implementación de aplicaciones en clúster contribuye a lograr muchos de estos objetivos. También es útil cuando la plataforma puede manejar tales cambios sin requerir que las aplicaciones sean rediseñadas, recodificadas o implementadas nuevamente.

## 1.4. Escenarios típicos de servicios web

1.4.1. Interactuar con socios comerciales
La empresa constructora de aventuras trabaja con una serie de socios para ayudarlo a proporcionar paquetes completos de aventuras a sus clientes. Estos socios van desde pequeñas empresas, como tiendas tipo "mamá y papá" que pueden organizar una actividad divertida en un lugar, hasta grandes empresas, como las principales aerolíneas y cadenas hoteleras. En efecto, estos socios cumplen con varios componentes de los paquetes de viaje solicitados por los clientes.

Además, existe una comunicación bidireccional entre el constructor de aventuras y sus socios. La empresa constructora de aventuras comunica varios pedidos a diferentes socios. Los socios, a su vez, facturan sus costos a la empresa constructora de aventuras. Es esencial que la empresa resuelva el manejo de esta comunicación entre socios cuando se conecta en línea.

Los servicios web pueden ser una forma ideal de integrar la empresa con múltiples socios por varias razones. Los servicios web son más rentables que el intercambio electrónico de datos (EDI), actualmente la solución más común disponible para la interacción con los socios comerciales. EDI requiere una inversión inicial significativa por parte de todas las partes. Si bien sus grandes socios comerciales pueden tener una infraestructura EDI extensa, la empresa constructora de aventuras no puede esperar que sus muchos socios comerciales pequeños gasten los recursos para implementar dicha infraestructura EDI. La interacción basada en servicios web es una buena solución para las pequeñas empresas que no tienen inversión en una infraestructura EDI y es más rentable para empresas más grandes con infraestructuras EDI existentes.

Además, los servicios web se pueden utilizar en una variedad de plataformas de hardware y software. Los socios existentes y potenciales no necesitan comprar o arrendar una plataforma de hardware o software en particular. Lo más probable es que los socios puedan usar sus sistemas existentes siempre que estos sistemas admitan servicios web. Dado que muchos de los líderes de la industria cooperan en los estándares y especificaciones de los servicios web, se espera que la mayoría de las plataformas soporten los servicios web.

La interacción basada en servicios web brinda a los socios, especialmente a los socios más grandes con importantes inversiones en infraestructura tecnológica, una ventaja adicional: pueden usar los servicios web para integrar sus sistemas empresariales existentes. Las empresas que han integrado sus sistemas existentes con los servicios web ofrecen una experiencia de interacción mejorada y más automatizada para socios como el constructor de aventuras. El resultado es una mejor experiencia del cliente, ahorro de costos y mayor interoperabilidad.

Los servicios web pueden satisfacer de manera eficiente esta necesidad de integrar tecnologías existentes con nuevas tecnologías. Dado que se basan en estándares universalmente aceptados e independientes de la plataforma, los servicios web constituyen una capa de integración natural. Ahora es posible proporcionar software para exponer un EIS existente como un servicio web. Esto hace factible que los usuarios que requieran acceso a ese EIS lo hagan a través de su servicio Web.

1.4.3. Llegar a diversos clientes
Los servicios web hacen posible que una empresa ponga su funcionalidad a disposición de varios tipos de clientes. La empresa constructora de aventuras, para mejorar la experiencia del cliente, puede proporcionar servicios como seguimiento de pedidos de clientes e información meteorológica en los puntos de destino del viaje, y la empresa puede garantizar que estos servicios sean accesibles para cualquier tipo de cliente. Los clientes pueden obtener la información que desean desde cualquier lugar, utilizando casi cualquier dispositivo. Algunos clientes pueden usar un navegador web para acceder a estos servicios, mientras que otros pueden preferir usar una interfaz más completa, como la de un cliente enriquecido escrito con tecnología J2SE o tecnologías de Microsoft. Otros clientes pueden usar dispositivos móviles, como teléfonos celulares o PDA, que se ejecutan en Java TM2 Entorno Platform Micro Edition (J2ME). La experiencia del cliente mejora cuando los usuarios finales tienen tal variedad de opciones para acceder a un servicio.

Los servicios web son una forma ideal de abrir estos servicios empresariales a todo tipo de clientes. Dado que exponen una interfaz estándar que es independiente de la plataforma y la tecnología, las funciones basadas en servicios web facilitan el desarrollo de clientes en cualquier plataforma que admita servicios web.

1.4.4. Agregación de datos de socios
Gran parte de las interacciones de la empresa constructora de aventuras con sus diversos socios comerciales se centra en obtener y agregar información de socios y otras fuentes dispares. Para proporcionar un paquete de aventura completo a sus clientes, el constructor de aventuras debe obtener de sus socios información como los detalles de los modos de viaje, alojamiento y actividades o aventuras. Esta información es dinámica por naturaleza, y la empresa constructora de aventuras necesita estas actualizaciones. Por ejemplo, el creador de aventuras necesita saber qué asientos están disponibles para una aerolínea en particular o qué hoteles tienen habitaciones disponibles y a qué precio. Esta información cambia con frecuencia y es importante que la empresa constructora de aventuras mantenga sus datos actualizados.

La naturaleza dinámica de la información de la empresa constructora de aventuras se extiende más allá de la disponibilidad actual de servicios. Es muy posible que uno o más de sus socios cambien sus servicios, o incluso que un socio comercial deje de existir. Para mantener la mejor experiencia posible para el cliente, la empresa constructora de aventuras debe poner a disposición los detalles del paquete más actualizados posibles. Para hacerlo, la empresa constructora de aventuras necesita mantener un catálogo dinámico de sus ofertas. Es decir, el creador de aventuras no solo crea su catálogo de paquetes de aventuras en función de los servicios disponibles de sus diversos socios, sino que también puede actualizar periódicamente su catálogo de aventuras, especialmente cuando se producen cambios con sus socios.

Los servicios web permiten una forma verdaderamente dinámica de construir y mantener este catálogo de información. Además, al incorporar sus servicios Web con servicios de registro, Adventure Builder puede ampliar su red de proveedores y permitir que participe cualquier número de proveedores o socios que elijan. Una vez que los proveedores señalan su participación a través del registro, el constructor de aventuras puede crear dinámicamente su propio catálogo a partir de las ofertas de los proveedores. Tenga en cuenta, sin embargo, que un arreglo verdaderamente dinámico requiere que exista un conjunto de acuerdos legales y financieros de antemano.

1.5.1. Descripción general de la plataforma J2EE
La plataforma J2EE está diseñada para proporcionar soporte del lado del servidor y del lado del cliente para desarrollar servicios web y aplicaciones empresariales distribuidas de varios niveles. Examinemos brevemente los conceptos principales que subyacen a la arquitectura de la plataforma J2EE. (Los lectores familiarizados con la plataforma J2EE pueden omitir esta sección).

modelo de varios niveles— La plataforma J2EE está diseñada para facilitar la implementación de servicios web y aplicaciones distribuidas de varios niveles. La plataforma define diferentes niveles, incluido un nivel de cliente, uno o más niveles intermedios y un nivel de back-end. También define una forma estándar para que estos niveles se comuniquen entre sí. Debido a que las aplicaciones pueden comunicarse de forma estándar, es posible que la lógica de una aplicación resida en diferentes niveles. Por lo tanto, las aplicaciones se pueden implementar en diferentes niveles de manera distribuida. Además, el modelo de varios niveles de la plataforma permite que varias partes de una aplicación se ejecuten en diferentes sistemas. A cada nivel también se le pueden asignar responsabilidades diferentes y distintas. Este modelo de desarrollo de varios niveles también permite que la plataforma brinde soporte enriquecido a una amplia variedad de clientes.

Desarrollo basado en componentes : la plataforma J2EE admite el desarrollo de aplicaciones y servicios basado en componentes. Las partes de la lógica de la aplicación que residen en niveles separados pueden hacer uso de los diferentes componentes proporcionados por la plataforma para cada nivel. Dicho desarrollo basado en componentes fomenta la reutilización, ya que los componentes son módulos discretos que prácticamente cualquier otro componente puede reutilizar cuando sea necesario. La plataforma puede admitir una variedad de componentes para representar la lógica comercial de la aplicación. Este tipo de modelo de desarrollo basado en componentes también permite distribuir las responsabilidades de desarrollo por conjunto de habilidades; es decir, asignar responsabilidades diferentes y distintas a varios desarrolladores en función de su conjunto de habilidades. Tal división por conjunto de habilidades ayuda a usar eficientemente las habilidades de los desarrolladores.

Gestión de componentes basada en contenedores— Para garantizar que los componentes interactúen de forma estándar, la plataforma introduce el concepto de gestión de componentes basada en contenedores. Los componentes se ejecutan en contenedores, que son entornos de tiempo de ejecución estandarizados que brindan servicios específicos a los componentes y, por lo tanto, garantizan la portabilidad de las aplicaciones en las implementaciones de la plataforma. En un entorno gestionado de este tipo, los contenedores se interponen en todas las llamadas a métodos y aplican sus servicios, que incluyen gestión de sesiones estándar, compatibilidad automatizada y declarable para transacciones de bases de datos, seguridad, acceso estandarizado a EIS, implementación estandarizada de aplicaciones (incluidos los medios para modificar el comportamiento de las aplicaciones en el despliegue), y así sucesivamente. Simplifica la arquitectura y el desarrollo : el modelo de desarrollo J2EE basado en componentes mejora la productividad del desarrollo de aplicaciones porque los componentes se corresponden bien con la funcionalidad deseada de la aplicación y el servicio que se va a desarrollar. Mediante el uso de herramientas, los desarrolladores pueden configurar el comportamiento de la aplicación en el ensamblaje o la implementación, según sea necesario, sabiendo que pueden confiar en la interacción estandarizada entre los componentes y los servicios estándar proporcionados por el contenedor para el componente.

Garantiza la compatibilidad con los estándares de servicios web emergentes : la plataforma J2EE sigue evolucionando en su compatibilidad con los estándares de servicios web y el perfil básico de WS-I. Esto garantiza que la lógica de la aplicación, ya sea nueva lógica o mejoras a la lógica existente, desarrollada con los componentes J2EE estandarizados, pueda exponerse fácilmente como servicios web.

Garantiza el desarrollo de servicios portátiles e interoperables : la plataforma J2EE, dado que es compatible con los estándares comunes J2EE y WS-I, garantiza no solo que las aplicaciones sean portátiles en las implementaciones J2EE, sino también que los servicios sean interoperables con cualquier servicio web implementado en cualquier otro. plataforma que cumple con los estándares WS-I.

Permite la integración con los sistemas de información existentes : la plataforma J2EE proporciona API estándar de la industria, como la arquitectura J2EE Connector, la API JDBC, Java Message Service, entre otros, para acceder a los EIS. Combinando estas API con su soporte para servicios web, la plataforma J2EE proporciona una excelente manera de integrar los EIS existentes y hacer que sus datos estén disponibles para los clientes en entornos de plataformas heterogéneas.

Es escalable para satisfacer las variaciones de la demanda : los contenedores J2EE proporcionan un mecanismo que admite el escalado simplificado de aplicaciones distribuidas con un esfuerzo de desarrollo de aplicaciones limitado. Los contenedores J2EE brindan soporte de transacciones, conexiones de bases de datos, administración del ciclo de vida y otros servicios que son escalables y no requieren código de los desarrolladores de aplicaciones.

Proporciona un modelo de seguridad flexible : el modelo de seguridad J2EE está diseñado para ser flexible, lo que permite a los desarrolladores de componentes especificar de forma declarativa los requisitos de seguridad de los componentes. Tanto la tecnología Enterprise JavaBeans como las API de Java Servlet también brindan control de seguridad mediante programación.

1.5.3.1. Tecnologías de componentes
Los desarrolladores utilizan las tecnologías de los componentes de la plataforma para crear las partes esenciales de una aplicación empresarial y un servicio web, como la lógica comercial del servicio. El enfoque de usar tecnologías de componentes para desarrollar las partes esenciales de un servicio mantiene el desarrollo modular y promueve la reutilización de módulos. Las diversas tecnologías de componentes de la plataforma J2EE son las siguientes:

Componente de cliente : la plataforma brinda soporte para diferentes tipos de clientes para interactuar con componentes en el lado del servidor. Los clientes Applet son clientes basados ​​en Java que generalmente se ejecutan desde un navegador web y tienen acceso total a las características de la plataforma J2SE. Los clientes de aplicaciones (a menudo denominados clientes autónomos) se ejecutan en sus propios contenedores y tienen acceso total a los servicios de la plataforma J2EE, como búsquedas JNDI, mensajería asíncrona, etc. Estos clientes pueden interactuar directamente con los componentes Web y EJB en el lado del servidor de la aplicación.

Componente web : los componentes web proporcionan una respuesta a una solicitud recibida a través de HTTP. La plataforma J2EE define dos tipos distintos de componentes web. Los componentes de servlet amplían la funcionalidad de un servidor web de forma portátil y eficiente. Con los servlets, los desarrolladores pueden asignar un conjunto de URL a un conjunto de servlets. Como resultado de dicha asignación, una solicitud HTTP a una de las URL invoca el servlet asignado, que a su vez procesa la solicitud y devuelve una respuesta. Los componentes de servlet también se pueden exponer como servicios web. Los componentes JSP , así como los componentes de servlet, permiten la generación de contenido dinámico.

Componente Enterprise JavaBeans : los componentes Enterprise JavaBeans (EJB) están diseñados específicamente teniendo en cuenta la lógica empresarial. Los componentes EJB son escalables, transaccionales y seguros. Los componentes del bean de sesión generalmente brindan servicios a un solo cliente y su estado no se puede recuperar después de un bloqueo del servidor. Además, los componentes de beans de sesión sin estado se pueden exponer como servicios web. Los componentes de bean de entidad son la representación de objetos de los datos mantenidos en un almacén de datos. Estos componentes gestionan datos persistentes, ya sea gestionando la persistencia por sí mismos o dependiendo del contenedor para gestionar su persistencia. Componentes de beans controlados por mensajespermitir que los clientes accedan a la lógica empresarial contenida en los componentes de bean empresarial de forma asíncrona. Un servicio de temporizador permite la implementación de operaciones temporizadas.

El uso de estas tecnologías de componentes para implementar una aplicación o servicio garantiza la estandarización de la aplicación o servicio. Además, dicha estandarización permite la reutilización y portabilidad de la aplicación y el servicio. Debido a que abordan diferentes aspectos de una aplicación o servicio empresarial, el uso de estos componentes ayuda a dividir el desarrollo en diferentes conjuntos de habilidades y hacer un uso eficiente de las diferentes habilidades.

1.5.3.2. Servicios de Plataforma y Contenedores
Las tecnologías de los componentes de la plataforma J2EE (los componentes de cliente, web y EJB) dependen del soporte del contenedor J2EE para funcionar correctamente. El estándar de plataforma J2EE, para garantizar que los componentes sean portátiles, requiere un proveedor de plataforma conforme para que ciertos servicios estén disponibles. Las aplicaciones empresariales y los servicios web creados con componentes portátiles y servicios estándar tienen asegurada la portabilidad. Entre los servicios requeridos de la plataforma se encuentran:

Servicio de nombres : un servicio de nombres permite el acceso simbólico a los recursos y componentes de EIS dentro de un entorno de nombres. Estos componentes se pueden personalizar cuando se ensamblan o implementan sin necesidad de cambios en el código fuente de búsqueda.

Servicio de implementación : un servicio de implementación permite cambios en el comportamiento de los componentes (como los requisitos de transacción y los requisitos de seguridad) en la implementación sin necesidad de cambiar el código fuente de un componente.

Servicio de transacciones : un servicio de transacciones libera al desarrollador de componentes de tener que incluir código para manejar problemas transaccionales como el acceso multiusuario y fallas/recuperaciones. Un servicio de transacción permite especificar los requisitos de transacción de los componentes cuando se ensamblan.

Servicio de seguridad : un servicio de seguridad garantiza que solo aquellos autorizados para acceder accedan a los componentes y recursos. Además, un servicio de seguridad proporciona autenticación y confidencialidad, entre otros servicios, para los usuarios.

1.5.3.3. Comunicación
La plataforma J2EE, además de especificar tecnologías de componentes y servicios de plataforma, también requiere un conjunto de tecnologías de comunicación estándar. Estas tecnologías de comunicación reúnen los componentes y servicios de la plataforma, lo que convierte a la plataforma J2EE en una plataforma estándar integrada para desarrollar servicios web y aplicaciones empresariales portátiles e interoperables.

Protocolos de Internet : la plataforma J2EE admite protocolos de Internet estándar y comunes como TCP/IP, HTTP, SSL, etc. Estos protocolos de Internet permiten la comunicación entre componentes y entre componentes y sus clientes.

Protocolos de invocación de métodos remotos (RMI) : la plataforma J2EE es compatible con Java RMI. Java RMI se basa en las API de invocación de método remoto, que utilizan interfaces de lenguaje Java para definir objetos de interfaz remota. La plataforma utiliza IIOP para convertir invocaciones de métodos locales en invocaciones de métodos remotos.

Tecnologías de mensajería : además de su compatibilidad con estos protocolos síncronos de Internet y RMI, la plataforma J2EE admite tecnologías que permiten la comunicación asíncrona. Ejemplos de estas tecnologías son la API de Java Message Service y la API de JavaMail.

Tecnologías de servicios web : la plataforma J2EE también admite tecnologías y protocolos específicos de servicios web que, junto con las tecnologías y protocolos ya mencionados, estandarizan la comunicación entre los componentes J2EE y los clientes J2EE. Con la llegada de los servicios web, que mejoran la interoperabilidad con clientes que no son de Java, la plataforma J2EE admite estándares de servicios web como SOAP y UDDI utilizando tecnologías como Java API para RPC basado en XML, Java API para registros XML, etc.

## 1.6. Conclusión

El desafío para los profesionales de TI y computación hoy en día es desarrollar e implementar eficientemente aplicaciones distribuidas y servicios web para su uso tanto en intranets corporativas como en Internet. Las empresas que pueden hacer esto de manera efectiva seguramente obtendrán una ventaja estratégica en la economía de la información.

Java 2 Platform, Enterprise Edition es un conjunto estándar de tecnologías Java que agilizan el desarrollo, la implementación y la gestión de servicios web y aplicaciones empresariales. La plataforma J2EE proporciona un entorno funcionalmente completo, en el que es posible desarrollar una gran clase de servicios web y aplicaciones empresariales utilizando tecnologías J2EE. Además, los desarrolladores pueden estar seguros de que las aplicaciones escritas para la plataforma J2EE se ejecutarán en cualquier servidor compatible con J2EE. En general, la plataforma J2EE brinda numerosos beneficios para las organizaciones que desarrollan estas aplicaciones de servicios web, incluido un modelo de desarrollo simplificado; escalabilidad de fuerza industrial; apoyo a los sistemas de información existentes; opciones en servidores, herramientas y componentes; y un modelo de seguridad simple y flexible.

Al brindar la capacidad de implementar aplicaciones cliente y puntos finales de servicios web orientados a componentes de una manera independiente de la plataforma, la plataforma J2EE brinda a las empresas de rápido movimiento una ventaja competitiva significativa y medible.
